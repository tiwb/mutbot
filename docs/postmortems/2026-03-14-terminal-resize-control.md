## 起源（773368ff）

触发事件：终端大小被一个**神秘的 1x1 像素窗口**改成了极小值，导致终端显示错误。AI 发现了这个隐藏窗口。

用户首次完整表达需求（**这是整个功能的种子**）：

> "关于terminal大小，可能确实需要保护机制，我的想法是找下最后活跃的终端，可能是有输入的，让窗口大小可以适配到这个client上。不过这个效果我不确定，所以先试试，然后再给一个功能，terminal的右键里增加一个标记Terminal大小跟随这个客户端面板的功能，用户手动操作，可以抢夺对terminal大小的控制权。再加一个自动的选项，自动开启后就是在这个终端有输入就自动抢夺。"

注意：这段话里**已经包含了最终方案的全部核心元素**：
1. "最后活跃的终端，可能是有输入的" → `_last_input_client`
2. "右键里增加一个标记，手动抢夺控制权" → `_follow_me` (Follow Me)
3. "自动的选项，有输入就自动抢夺" → Auto 模式

## 第一次迭代（35204be9）

**设计方案**：`_primary_client` + `_primary_locked` + `claim_resize`
- 一个 primary 状态（谁在控制），一个 locked 标记（是否手动锁定）
- `claim_resize` 消息手动锁定

**测试发现的问题**：
- 较小屏幕刷新时不跟随大尺寸 → 非主客户端显示错误
- 新连接没有获取当前 PTY 尺寸 → 显示折行
- 手机端溢出内容被剪裁
- AI 自信地直接改代码，用户叫停："不要那么自信的直接修改代码，先给我你的修改方案"
- 发现长 scrollback 导致闪烁 → 引出 pyte 方案
- **手机无法触发右键菜单** → Follow Me 功能在手机上不可用

## 第二次迭代（0b430346）

用户测试第一次实施后发现根本性设计问题：

> "终端右键菜单里面的 Follow input 和 Follow me 状态似乎没有同步。PC 端已经 follow 了，但手机连接上来时抢占了大小。"

10+ 轮讨论的**关键转折点**：

| 轮次 | 用户的认知演进 |
|------|--------------|
| [11] | "只需要一个信息——谁是尺寸控制者" → 试图简化为单一状态 |
| [12] | "还是需要表达'没有 resize owner'的概念" → 发现不能太简化 |
| [13] | **核心突破**："两种状态。一种随最后输入者，另一层用户主动锁定" |
| [14] | "Active client 会让人误解，叫 Last input client" → 命名定型 |
| [14] | "两个菜单项比较好，一个 Auto，一个 Follow" → 交互确定 |

最终方案：`_follow_me` + `_last_input_client` 两层模型 — 与用户在**起源会话**中的直觉描述完全一致。

## 核心洞察

1. **用户的第一句话就是正确答案**：起源会话里的三点描述（最后输入者 + 手动抢占 + 自动模式）与最终实现完全对应。但中间经历了"简化 → 发现不够 → 回到原始设计"的过程。

2. **第一次迭代的问题不是需求理解错误，而是模型抽象错误**：用 `_primary_client` + `_primary_locked` 来表达用户的三个需求，导致语义不清、状态纠缠。第二次用 `_follow_me` + `_last_input_client` 把两层正交分离，问题自然消解。

3. **命名是设计的核心**：从 "primary" → "active client" → "last input client" 的演进，每次命名修正都推动了设计清晰化。"Last input client" 这个名字精确到无歧义。

4. **实施暴露设计问题**：第一次迭代的设计在纸面上看起来合理，但实际测试（手机连接抢占、状态不同步）暴露了模型的根本缺陷。

## 方法论讨论

### 原型驱动的需求发现

复盘初期，AI 的第一反应是"加流程"——设计阶段增加命名审查、状态推演、场景验证。用户否定了这个方向：

> "冗余的流程只会让我们做事情被束缚。快速地迭代出原型，然后针对原型进行体验修改。用户才能慢慢地知道自己的需求。"

关键认知：**用户通过体验原型来发现自己的真实需求，而不是通过思考来提前定义需求**。第一次迭代不是"浪费"，它是用户认知过程的必要步骤。原型给了用户一个具体的东西去"不满意"，而"不满意"比"满意"更容易表达清楚。

真正的路径是：

```
用户感受 → AI 快速实现 → 用户体验 → 用户产生更精确的不满 → AI 帮助用户把"不满"转化为"设计约束" → 迭代
```

AI 最有价值的角色不是在第一步"理解需求"，而是在"不满"到"设计约束"的转化环节。用户说"手机不应该抢占"，AI 需要识别出这意味着"连接时不应该自动获得控制权，控制权的获取必须是显式操作"。

### 需求与不满：两个约束面

用户提出的核心观点：

> "不满在很多情况下和需求本身一样重要，它是需求的两个约束面。需求是开放的、无限的。不满是具体的。我们应该从两个方面同时去入手，慢慢的收敛设计。"

需求描述的是"想要什么"（开放、无界），不满描述的是"不能接受什么"（具体、有界）。两者共同构成设计的约束空间。单靠需求无法收敛，单靠不满无法构建。

更关键的是：**需求和不满本身都可能反转、都可能是错的**。在第二次迭代中，用户先说"只需要一个信息就够了"（信号 C），随后自己推翻了这个判断，说"还是需要表达没有 owner 的概念"（信号 D）。AI 不应该盲目执行最新的信号，而应该在所有已积累的信号中寻找最一致的解释。

### AI 作为持续推断引擎

用户对 AI 角色的定位：

> "AI 应该做的是收集这些信息，然后不断地在所有的信息之间进行计算，给出一个最合理的实施路径，而不是在一个具体的细节问题上面纠结。"

回到 resize 的信号序列：

| 信号 | 内容 | 方向 |
|------|------|------|
| A | "最后活跃的终端，可能是有输入的" | 两层模型 |
| B | "手机不应该抢占" | 显式操作 |
| C | "只需要一个信息就够了" | 单一状态 |
| D | "还是需要表达没有 owner 的概念" | 推翻 C |
| E | "两种状态：随最后输入者 + 主动锁定" | 两层模型 |

A、B、D、E 是收敛的，C 是离群值。如果 AI 在信号 D 出现时就能说："结合你之前的描述，两层模型更符合你的整体意图"，收敛会更快。

这改变的不是 SDD 的某个步骤，而是 AI 在整个协作过程中的角色定位。

### 执行者与推断者

最终结论：**AI 既要是执行者，也要能成为推断者**。

- 作为**执行者**：快速实现原型，不纠结于"是否完全理解了需求"，让用户有东西可以体验和反馈。
- 作为**推断者**：在所有已积累的信号（需求 + 不满 + 反转）中持续计算最一致的解释，当新信号与历史信号矛盾时，判断哪个更可能是噪声，主动帮助用户收敛。

这两个角色不是交替的，而是并行的。AI 在执行的同时推断，在推断的同时执行。这是否能做到，以及如何做到，是一个开放问题。
