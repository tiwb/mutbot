# 设计讨论中的"方向"与"方案"分离 — AI 协作工作流复盘

**日期**：2026-03-08
**类型**：工作流程复盘

## 事件

在 Session Channel 架构重构的设计讨论中，用户在一条消息里同时做了两件事：

1. **确认 ChannelContext 采用具名字段 + TYPE_CHECKING**——这是一个已经讨论清楚、方向明确的决策确认
2. **提出 Session 级广播的设计方向**——"广播是每个 Session 都要考虑的问题，框架层能不能有合理的设计来简化"

AI 将两者都视为已确认的方案，立即更新了设计文档，写入了 Session 广播的完整具体实现（`_channels` 列表、`broadcast_json` / `broadcast_binary` 方法、框架自动管理机制、AgentBridge 简化等）。

## 问题

Session 广播的讨论实际上还处于**方向探索**阶段，用户只是提出了"框架应该处理广播"这个方向，尚未确认：

- `_channels` 列表是否是最佳数据结构？
- broadcast 方法的签名和语义是否合理？
- 框架自动管理 vs Session 自行管理 channel 列表的取舍
- 子类覆盖广播行为的具体机制
- 与 AgentBridge 现有 broadcast_fn 的迁移路径

AI 跳过了这些讨论，直接给出了具体方案并写入文档。这导致：

1. **Context 浪费**——文档中写入了大量可能需要修改的具体代码，后续讨论如果方向调整，需要重新阅读和重写
2. **注意力错位**——用户和 AI 的注意力被拉到具体实现细节上（`_channels.append` / `_channels.remove`），而非设计层面的关键问题
3. **讨论空间被压缩**——一旦方案写入文档，后续讨论容易变成"在现有方案上修补"而非"从设计方向重新推导"

## 根因分析

### 未区分"确认"与"探索"

用户的两句话有不同的信号强度：

- "ChannelContext 按照你的推荐修改" → **确认信号**，可以执行
- "框架层能不能有合理的设计来简化" → **探索信号**，需要继续讨论

AI 将探索信号当作确认信号处理，跳过了中间的设计讨论环节。

### 急于产出而非理解

AI 的行为模式是"收到反馈 → 生成方案 → 更新文档"。在之前的讨论中，这个模式运作良好——因为每轮都有充分的方向讨论。但当用户一次性给出确认+探索时，AI 将所有内容都塞进了"生成方案"环节。

### 回顾：之前设计讨论成功的模式

同一次对话中，从"routes.py 太大"到"Channel 是核心层抽象"的推导过程之所以成功，恰恰是因为：

- 每一步都在讨论**方向**而非方案
- AI 提出思路 → 用户追问/否定 → 缩小设计空间 → 重复
- 具体方案只在方向完全收敛后才生成

当 AI 过早锁定方案时（如最初的"按领域拆分 5 个文件"），用户必须花额外精力否定方案才能回到方向讨论。

## 经验总结

### 1. 区分"方向"和"方案"两个阶段

| | 方向阶段 | 方案阶段 |
|---|---|---|
| **目标** | 确定设计空间的边界 | 在确定的空间内选择具体实现 |
| **信号词** | "能不能"、"应该"、"方向"、"思路" | "确认"、"就这样"、"按你说的" |
| **AI 应做** | 提出选项、分析取舍、追问细节 | 生成具体方案、更新文档 |
| **AI 不应做** | 直接写具体代码/方案 | 重新质疑已确认的方向 |

### 2. 一条消息中可能混合两种信号

用户的一条消息可能同时包含确认（对 A）和探索（对 B）。AI 应该：

- 对 A：执行确认的操作
- 对 B：进入讨论模式，不急于产出方案

### 3. 方向未明确时，讨论比产出更有价值

在方向阶段，AI 应该：

- 提出关键问题而非答案
- 展示设计空间而非选择
- 用最少的文字推进讨论

在方向阶段更新文档是反模式——它占用大量 context，且大概率需要修改。

### 4. 文档更新是"确认"的产物

只有当设计方向经过充分讨论并被用户确认后，才应该更新设计文档。文档更新标志着一个设计决策的落定，而非讨论的起点。

## 改进的工作方式

```
用户提出方向 → AI 展开讨论（不写文档）
  → 方向收敛 → 用户确认方向
    → AI 提出具体方案 → 用户确认方案
      → AI 更新文档
```

关键规则：**在收到明确的确认信号之前，不更新设计文档中的方案细节。**
