# Mobile Connect 本地地址二维码 BUG 复盘

**日期**：2026-03-30
**类型**：Debug 过程复盘
**修复**：`App.tsx` 桌面布局 `MobileConnectDialog` 漏传 `local` prop

## BUG 描述

访问 `127.0.0.1:8741` 时，点击 Mobile Connect 菜单应显示"仅本地可访问"的配置提示，实际仍显示二维码。PC 上 100% 复现。

## 根因

`App.tsx` 有三段式 return（早期 return → 移动布局 → 桌面布局）。在前一个会话的 MobileConnect 改造中，`replace_all` 因缩进差异未能替换所有调用点，手动补时桌面布局的 `<MobileConnectDialog>` 漏传了 `local` prop。

```tsx
// 移动布局（第 1267 行）— ✅ 有 local
<MobileConnectDialog url={...} via={...} local={mobileConnectData.local} onClose={...} />

// 桌面布局（第 1357 行）— ❌ 漏了 local
<MobileConnectDialog url={...} via={...} onClose={...} />
```

修复：加上 `local={mobileConnectData.local}`，重新构建前端。

## BUG 引入过程（前一个会话）

MobileConnect 从多地址列表改为单地址模式时：
1. `replace_all` 替换 `MobileConnectDialog` 的 props，但两处缩进不同，第二处未被替换
2. 手动补了第二处的 props，但漏加了 `local`
3. 编译通过，未验证所有调用点是否一致
4. 会话中已意识到问题并尝试修复，但修复不完整

**引入原因**：实现完成后缺少独立验证步骤（如 grep 所有调用点核对 props）。

## Debug 过程（当前会话）

### 第一轮分析（30 秒，正确定位）

grep `MobileConnectDialog` → 发现两处调用 → 一处有 `local` 一处没有 → **BUG 已定位**。

### 误判（关键转折点）

根据行号和注释，将第 1267 行判定为"桌面布局"、第 1357 行判定为"移动布局"。**未读代码结构验证这个假设。**

用户反馈"我在 PC 上发现的"，与"移动布局漏传"的结论矛盾。此时正确做法是验证假设（读 App.tsx 的 return 结构，确认哪个是桌面布局），但 AI 选择了放弃已有定位，转向别的方向。

### 无效排查（15 分钟）

| 排查方向 | 结果 | 必要性 |
|---------|------|--------|
| 后端 Client.origin 检测 | 正确 | 不必要 |
| 前端构建产物是否最新 | 是最新 | 不必要 |
| 浏览器缓存 | JS hash 一致 | 不必要 |
| 构建产物中 local 字段处理 | 逻辑正确 | 不必要 |
| 注入后端 debug wrapper | 确认返回 `local: true` | 不必要 |
| 注入前端 MutationObserver | 确认走了 QR 分支 | 间接有用 |
| React fiber 分析 props | 确认 `local` 未传入 | 间接有用 |
| React fiber 分析 state | 确认 state 中 `local: true` | 间接有用 |

所有间接验证的结论汇总后，指向同一个结果：调用方没传 `local` prop —— 即第一轮分析的结论。

### 最终定位

读 App.tsx 的 return 结构，发现三段式 return。第 1267 行在移动布局 return 中（有 `local`），第 1357 行在桌面布局 return 中（漏 `local`）。与最初定位一致，只是桌面/移动搞反了。

## 模式分析

### 与 Resize 控制权复盘的共性

| | Resize 设计 | Mobile Connect Debug |
|---|---|---|
| 正确答案出现时机 | 用户第一句话 | 第一轮 grep |
| 干扰信号 | "只需要一个信息就够了" | "我在 PC 上发现的" |
| AI 的反应 | 默默执行最新指令 | 放弃定位，转向运行时调试 |
| 正确反应 | 指出矛盾，坚持全局信号 | 验证假设（读代码结构） |
| 绕行代价 | 多走一轮设计迭代 | 15 分钟无效排查 |

**共同模式**：已有正确结论 → 出现矛盾信号 → AI 未验证矛盾本身是否成立 → 放弃正确结论 → 绕一大圈回到原点。

### 新的观察：新会话 vs 同一会话

- **同一会话**（实现会话）：积累了大量实现上下文，容易"灯下黑"——编译通过就以为做对了
- **新会话**（debug 会话）：没有实现包袱，直接看代码现状，30 秒就定位了 BUG

**但新会话也有陷阱**：找到了正确答案后，被一个矛盾信号干扰，又花了 15 分钟绕回来。问题不在于"能不能找到"，而在于"找到后能不能坚持"。

### 静态 BUG 与动态调试

这是一个**静态可分析**的 BUG（少传了一个 prop），完全不需要运行时调试。但 AI 在放弃静态分析结论后，转向了复杂的运行时手段（注入 Python wrapper、JS observer、React fiber 分析），用动态方法解决静态问题，大幅增加了时间成本。

**判断标准**：如果 BUG 可以通过读代码确认，就不需要运行时调试。运行时调试适用于状态依赖、时序问题、环境差异等静态分析无法覆盖的场景。

## 经验总结

1. **假设决定方向时，转向前必须验证假设**。"第 1267 行是桌面布局"这个假设决定了后续所有行动方向，验证成本是读 10 行代码（30 秒），不验证的代价是 15 分钟无效排查。

2. **实现完成后，独立验证所有变更点**。`replace_all` 失败后手动补，应该 grep 一下确认所有调用点 props 一致。编译通过不等于逻辑正确。

3. **静态 BUG 用静态方法**。少传一个 prop 是读代码就能确认的事，不需要注入运行时调试工具。

4. **新会话审视实现结果比同一会话自检更可靠**。实现会话有认知偏差（"我做过了所以应该对了"），新会话没有这个包袱。
