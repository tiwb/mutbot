# 终端可靠性修复回顾与工作方式改进

**日期**：2026-03-08
**类型**：回顾 + 流程规范

## 修复经过

### 阶段一：原始 Bug 修复（顺利）

**目标**：修复 `bugfix-terminal-reliability.md` 中的两个 bug。

**Bug 1 — Scrollback 持久化缺失**：PTY 退出时 scrollback 留在内存、不写盘。

- 修复：`on_dead` 回调 + `mark_dirty` + 定时持久化
- 顺利完成，无 regression

**Bug 2 — 多客户端尺寸输出混乱**：PTY 缩小时大窗口看到错乱输出。

- 修复：服务端广播 `pty_resize`，前端 `term.resize()` 匹配 PTY 尺寸
- 顺利完成

**阶段一成功的原因**：问题边界清晰，改动范围小，有设计文档指导。

### 阶段二：测试中发现的衍生问题（开始偏离）

测试中陆续发现新问题，进入修一个、出一个的循环：

1. **CSS 背景色** — xterm.js v6 的 `.xterm-viewport` 遗留元素。修复简单，`display: none`。✓
2. **重启幂等性** — 两窗口同时 `session.restart` 创建重复 PTY。修复：alive 检查。✓
3. **Channel 生命周期** — 终端重启不应断 channel。修复：不关闭 channel + re-attach。✓ 但引入了 RAF 时序 bug。

### 阶段三：RAF 时序 Bug（反复修改的开始）

**问题**：`scrollback_done` 用 RAF 延迟执行，`process_exit` 同步执行。对于死终端，两条消息先后到达，RAF 回调把 `expired` 重置为 false → Restart 按钮消失。

**修复过程**：

| 轮次 | 做了什么 | 结果 |
|------|---------|------|
| 1 | 用 `ready` 消息替代 `scrollback_done` + `process_exit` | ✓ 消除时序问题 |
| 2 | 同时把 `init()` 改为始终调 `session.restart` | ✗ 破坏"死终端展示历史"设计 |
| 3 | 回退 `init()` | ✓ 但已浪费一轮 |
| 4 | 添加 `_strip_replay_queries` 过滤 scrollback 中的 DA1 | ✗ 实际 DA1 来自 live 输出 |
| 5 | 前端 `onData` 过滤终端响应 | ✓ 从 log 确认根因后的正确修复 |
| 6 | 前端 `processExited && alive` 判断清屏 | ✗ 遗漏 `ready(alive=false)` 不设 `processExited` 的路径 |
| 7 | 补设 `processExited = true` | ✗ 修补越来越复杂 |
| 8 | **服务端始终发清屏 + scrollback** | ✓ 根本性简化 |

**轮次 2-7 全部是无效工作**。最终解法（轮次 1 + 5 + 8）如果一开始就想清楚，只需要 3 步。

## 根因分析

### 为什么会反复修改？

**根本原因不是技术问题，而是思维模式问题。**

#### 1. 在代码层面思考，而非在架构层面思考

每次遇到 bug，立即跳到"哪行代码需要改"，而不是先问"这个系统的职责划分对不对"。

典型案例：前端用 `processExited` 标志位判断是否清屏。这本身就是错的——清屏是后端的职责，前端不应该推理。但因为在代码层面修补，每次只看到"这个 flag 漏设了"，修了一个 flag 又漏另一个。

**教训**：遇到复杂的条件判断时，不要继续加条件，而是退一步问：**这个判断应该在这里做吗？** 谁拥有做这个决定的完整信息？

#### 2. 基于推理修 bug，而非基于数据

假设"extra characters"来自 scrollback 回放中的 DA1 查询。花时间写了 `_strip_replay_queries`。实际从 log 看，DA1 来自新 PTY 的 live 输出，scrollback 过滤根本不在问题路径上。

**教训**：**先看 log，再写代码。** 推理可以形成假设，但假设必须通过 log 数据验证后才能写修复。

#### 3. 一次改多件事

轮次 2 同时做了三件事：`ready` 消息替代、`init()` 简化、RAF → `term.write` 回调。其中一个引入 regression，但三个混在一起无法定位。

**教训**：**一次只改一件事，测完再改下一件。** 即使两个改动看起来"顺手一起做"，也要分开。

#### 4. 把"减少代码路径"当作简化

认为"始终 `session.restart` 就不需要死终端路径了"。代码确实少了一条路径，但删掉了一个业务功能。

**教训**：**真正的简化是让正确的组件承担正确的职责**，而不是减少代码行数。服务端始终发清屏才是真正的简化——职责清晰，前端不需要判断。

## 最终方案的设计原则

回顾最终正确的方案，它遵循了几个简单原则：

### 原则 1：后端驱动，前端渲染

后端知道所有上下文（是新建、重连、re-attach、死终端），由后端决定发什么数据。前端只负责写入 xterm 和响应状态消息。

- 清屏：后端发 `\x1b[2J\x1b[H`，前端不判断
- 状态：后端发 `ready(alive=true/false)`，前端据此切换 UI
- 退出：后端发 `process_exit`，前端显示 Restart 按钮

### 原则 2：一条消息表达一个意图

旧设计用 `scrollback_done` + `process_exit` 两条消息组合表达"死终端"，消息顺序和异步时序导致 bug。新设计用一条 `ready` 消息直接告知状态。

### 原则 3：过滤在正确的层做

- 终端响应（DA1 等）：前端 `onData` 过滤。这是 xterm.js 生成的数据，不应作为用户输入。与 scrollback 还是 live 无关。
- CSI 查询序列：服务端 scrollback 回放时过滤。防止回放触发响应。但这是防御层，主要保护靠前端过滤。

## 未来工作方式

### 遇到 Bug 时

1. **复现 → 看 log → 定位数据流**。不要跳过这一步直接改代码。
2. **问"这个判断应该在哪做"**。如果前端需要复杂状态推理才能决定行为，多半是职责划分错了。
3. **一次改一件事**。改完跑一遍完整场景。

### 遇到复杂条件时

如果发现自己在写 `if (flagA && flagB && !flagC)` 这样的代码，停下来问：

- 这些 flag 的组合是否穷举了所有场景？
- 是否有更上游的地方可以消除这些 flag 的必要性？
- 是否可以让数据的发送方（通常是后端）直接携带足够信息，接收方不需要推理？

### 改动前的检查

```
[ ] 从 log/数据确认了 bug 的实际路径（非纯推理）
[ ] 这次只改一件事
[ ] 改动不把判断逻辑放到错误的层（前端 vs 后端）
[ ] 列出受影响场景，改后逐一验证
```
